## Week 2: Hands-On

### First Code Change

Pick a small task to get comfortable with the workflow:

1. **Find an issue** labeled `good first issue` in GitHub
2. **Create a branch**: `git checkout -b feature/your-feature`
3. **Make your change**
4. **Run checks**: `npm test && npm run build`
5. **Submit a PR**

### Database Workflow

Understand how schema changes work:

```bash
# See current database state
npm run supabase:status

# Reset to clean state (applies migrations)
npm run supabase:reset

# Load seed data
npm run db:import-seed

# Save your current data (before experiments)
npm run db:save-snapshot

# Restore from snapshot
npm run db:restore-snapshot
```

<Callout type="warn">
Schema changes use a pull-based workflow. Changes are made to the remote TEST database first, then pulled down as migrations. See [Database Workflow](/docs/developer-guide/database-workflow) for details.
</Callout>

## Coding Standards

### File Size Limit

Files should be **under 500 lines**. If a file grows larger, split it into modules.

### TypeScript Rules

- **No `any` types** - use proper type definitions
- **Strict mode** - all strict checks enabled
- **Zod for runtime validation** - don't trust external data

### Component Patterns

```typescript
// Good: Typed props with interface
interface PartCardProps {
  part: Part;
  onSelect: (id: string) => void;
}

export function PartCard({ part, onSelect }: PartCardProps) {
  // ...
}
```

### API Patterns

```typescript
// Good: Zod validation at API boundary
import { partSchema } from '@/lib/schemas/part';

export async function POST(request: Request) {
  const body = await request.json();
  const result = partSchema.safeParse(body);

  if (!result.success) {
    return Response.json({ error: result.error }, { status: 400 });
  }

  // result.data is now typed and validated
}
```
